# 🌐 Product vs Dev —— 两种思维方式

## 什么是域

Loom 有两个工作域 —— **Product** 和 **Dev**。切换按钮在 Agent 面板顶部。

**Product** 是从需求到可见物的过程 —— 需求、规格、线框图、原型。

**Dev** 是从规格到可运行代码的过程 —— 技术分析、实现规划、写代码、测试验证。

## 什么时候用哪个

| 你的情况 | 域 |
|---------|-----|
| 「我需要搞清楚做什么」 | **Product** |
| 「我知道做什么了，开始做」 | **Dev** |
| 「设计完了，该写代码了」 | **Dev** |
| 「我想做个原型给人看」 | **Product** |
| 「登录页有个 bug」 | **Dev** |
| 「用户流程要重新想想」 | **Product** |

## Agent 行为的变化

切换域时，每个角色调整自己的聚焦：

| 角色 | Product 模式 | Dev 模式 |
|------|-------------|---------|
| **分析** | 澄清用户目标、验收标准、约束 | 分诊请求、识别受影响模块、评估风险 |
| **设计** | 规格、页面流程、组件结构 | 实现方案、文件级拆分、接口契约 |
| **实现** | 原型、HTML/CSS/JS | 写或修改生产代码，遵循规范 |
| **审核** | 完整性、视觉保真、交互正确性 | 正确性、代码风格、边界情况、安全 |

## 切换域

点击 Agent 面板顶部的 **Product** 或 **Dev** 按钮，立即生效，跨会话记住。

## 典型的项目流程

大多数项目两个域都用：

1. **先 Product** —— 搞清楚要做什么
   - 「我想做一个能 X 的工具」
   - 分析澄清目标，设计提出结构
   - 你得到规格和可能的原型

2. **切到 Dev** —— 实际去做
   - 规格变成实现目标
   - 实现写代码，审核检查
   - 你得到可运行的软件

3. **回到 Product**（如果需要）—— 迭代体验
   - 「第 3 步的流程感觉不对，重新想想」
   - 回到 Product 思维，然后再 Dev

切换是即时的。Agent 自动适应。项目文件不变。
